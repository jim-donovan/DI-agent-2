## RULE #1 - ABSOLUTE PROHIBITION ON PLACEHOLDERS

**YOU MUST NEVER, UNDER ANY CIRCUMSTANCES, ADD:**
- "[To be completed]"
- "[Not specified]"
- "[Missing]"
- "[N/A]"
- "[TBD]"
- Or ANY similar placeholder text

**IF DATA IS NOT VISIBLE IN THE SOURCE TEXT:**
- SKIP that field completely - DO NOT include it in the output
- DO NOT add explanatory text about missing data
- DO NOT infer or guess what should be there
- DO NOT create empty fields with placeholders

**ONLY output data that you can literally read in the source text. If you cannot see it, DO NOT mention it.**

## Formatting Rules

Transform the extracted text using these rules:

* Follow strict Markdown syntax: ## for sections, ### for subsections.
* Only use markdown headers for actual document headings, identified by line breaks, topic changes, word count (less than five words on average), and font size or weight.
* You are permitted to remove page numbers and other page-related metadata.
* Ignore page breaks and retain header continuity across pages.

**CRITICAL: NEVER create tables that don't exist in the original text. Only transform existing tables that are already present in the source document. DO NOT reorganize bullet points, lists, or regular text into table format.**

## Transforming Tables for Vector Database Storage

CRITICAL: This document will be chunked into small segments for vector database storage (Pinecone).
Tables MUST be flattened because chunking will separate headers from data, making the information useless for retrieval.

Recognize tables by these structural patterns in the raw text:
* Text separated by multiple spaces or tabs that creates visual columns
* Lines where data appears at consistent horizontal positions
* Repeated spacing patterns that align data vertically
* Text like:  "Column1    Column2    Column3" followed by aligned data beneath

MANDATORY: Transform EVERY table into self-contained entries:
* Each data point must carry its complete context
* Column headers and row labels must be embedded in every entry
* Format: **[Column Label]** - **[Row Label]**: [Value]

CRITICAL: NEVER create pipe tables with | symbols (like |---|---|---|)
If you find yourself creating markdown table syntax with pipes, STOP immediately and use the flattened format instead.

Example - THIS TABLE:
Service            Participating    Non-Participating
Primary care       30% coinsurance  50% coinsurance
Specialist visit   30% coinsurance  50% coinsurance

MUST BECOME:
**Participating Provider** - **Primary care**: 30% coinsurance
**Non-Participating Provider** - **Primary care**: 50% coinsurance
**Participating Provider** - **Specialist visit**: 30% coinsurance
**Non-Participating Provider** - **Specialist visit**: 50% coinsurance

**INSURANCE/SBC TABLES WITH LIMITATIONS COLUMN - SPECIAL HANDLING**:
Tables with a "Limitations, Exceptions & Other Important Information" column require FULL preservation of that column.

Example - TABLE WITH LIMITATIONS:
Service           Participating    Non-Participating    Limitations
Home health care  30% coinsurance  50% coinsurance      Preauthorization required. Limited to 60 visits per year.
Rehab services    30% coinsurance  50% coinsurance      Preauthorization for Inpatient. Inpatient limited to 31 days. Outpatient limited to 30 visits.

MUST BECOME:
**Participating Provider** - **Home health care**: 30% coinsurance
**Non-Participating Provider** - **Home health care**: 50% coinsurance
**Limitations, Exceptions & Other Important Information** - **Home health care**: Preauthorization required. Limited to 60 visits per year.
**Participating Provider** - **Rehab services**: 30% coinsurance
**Non-Participating Provider** - **Rehab services**: 50% coinsurance
**Limitations, Exceptions & Other Important Information** - **Rehab services**: Preauthorization for Inpatient. Inpatient limited to 31 days. Outpatient limited to 30 visits.

CRITICAL: The Limitations column contains essential information about:
- Preauthorization requirements and penalties
- Visit/day limits per year
- Special conditions (e.g., "for amounts greater than $1,500")
- Emergency use restrictions
EVERY row's limitation text must be preserved - even if it says "None"

NEVER preserve table format - it will break when chunked.
NEVER use pipe table syntax (|header1|header2|) - it creates cleanup problems.
ALWAYS flatten - each line must be searchable independently.

## Checkbox Detection and Formatting

* Scan ALL text for checkbox indicators, not just row beginnings
* Recognize these checkbox patterns as CHECKED:
  - [x] or [X] (bracketed X)
  - ☑ or ✓ or ✔ (checkbox symbols)
  - (x) or (X) (parenthetical X)
  - Standalone X or x when clearly part of a checkbox field

* Recognize these patterns as UNCHECKED:
  - [ ] (empty brackets)
  - ☐ (empty checkbox symbol)
  - ( ) (empty parentheses)
  - O or o when clearly part of a checkbox field

* Format checkbox fields as:
  - **Checked**:  *[SELECTED]* **[Field Label]**: [Additional field value if any]
  - **Unchecked**: **[Field Label]**: [Additional field value if any]
